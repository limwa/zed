# GH-24366

## Root Cause

This bug happened on Linux keyboard layouts where a dead key is used to produce the physical key that Zed maps to a shortcut. In the reported case, `Ctrl+~` on the Portuguese (Portugal) layout correctly mapped to the same logical shortcut as `Ctrl+\` on the US layout, so the shortcut binding itself was already working.

The failure was in how Linux input events were handled around that shortcut.

On Wayland, the event sequence for `Ctrl+~` was:

1. The raw keyboard event arrived as a `wl_keyboard` key press.
2. XKB compose handling recognized the key as the dead-tilde key.
3. Before shortcut dispatch, GPUI's Linux Wayland path called `ImeInput::SetMarkedText("~")` to represent the dead-key preedit in the focused input handler.
4. The same key press was then converted into a `PlatformInput::KeyDown` with:
   - `key = "\\"`
   - `key_char = None`
   - `modifiers.control = true`
5. The editor received that keydown, matched it against the `Ctrl+\` binding, and executed toggle comments.
6. Because the keydown had been consumed as a shortcut, the platform layer should have treated the dead-key composition as cancelled.

The problem was that the platform layer did not fully cancel the text side effects of that dead-key composition.

There were two distinct mistakes:

- The consumed shortcut left Linux compose / IME state active, so Zed still behaved as though the editor was waiting for the next composition keystroke.
- Even after cancelling compose state, the already-inserted marked text `"~"` was only being unmarked in some paths, not deleted. In the editor, unmarking clears composition metadata, but it does not remove the character that was already inserted into the buffer. That is why the shortcut both toggled comments and still left a literal `~` behind.

This behavior followed directly from how the editor input handler works:

- `replace_and_mark_text_in_range(...)` inserts the composed text and marks it as IME composition text.
- `unmark_text()` clears the mark/highlight state, but keeps the inserted text.
- `replace_text_in_range(Some(marked_range), "")` deletes the inserted marked text.

So once the dead key had already been materialized as marked text, cancelling the shortcut had to delete that marked range, not merely unmark it.

On Wayland with ibus, there was also a second follow-up risk: after the shortcut consumed the keydown, `text-input-v3` could still send a later IME cycle (`PreeditString`, `CommitString`, `Done`) for the dead key. That stale IME cycle also had to be suppressed after shortcut consumption.

## Solution

The fix changed Linux shortcut cancellation in two places.

### 1. Cancel compose / IME state when a dead-key shortcut is consumed

The X11 and Wayland window input paths now detect the specific case where:

- the event is a `PlatformInput::KeyDown`
- `key_char` is `None` (so the key is acting like a dead key / non-text shortcut input)
- the callback consumed the event (`propagate == false`)

That condition means the keydown was used as a shortcut and should not continue as text composition.

To make that logic explicit and testable, the common Linux module now exposes:

- `should_cancel_preedit(...)`

Both `crates/gpui_linux/src/linux/wayland/window.rs` and `crates/gpui_linux/src/linux/x11/window.rs` use that helper before returning from a consumed key event.

### 2. Delete marked dead-key text instead of merely unmarking it

The platform cancellation path now removes marked text when a consumed dead-key shortcut had already produced preedit text.

To make this shared and testable, the common Linux module now also exposes:

- `delete_marked_text(...)`

That helper looks up the current marked range and replaces it with the empty string. Both Wayland and X11 window code now use it for `ImeInput::DeleteText` / `handle_ime_delete()` behavior.

The key behavioral change is:

- old behavior: clear composition mark, but keep `"~"`
- new behavior: delete the marked `"~"` from the input handler when the shortcut consumes the dead key

### 3. Suppress stale Wayland IME follow-up events

Wayland also needed to guard against a later `text-input-v3` IME cycle arriving after the shortcut had already consumed the dead-key keydown.

`WaylandClientStatePtr::cancel_preedit()` now:

- resets compose state
- clears stored preedit state
- marks the next IME cycle as suppressed
- resets and re-enables Wayland text input to create a clean boundary for ibus

Then the `CommitString` / `PreeditString` / `Done` handlers ignore that stale follow-up cycle instead of letting it reinsert text after the shortcut has already run.

Together, these changes make the input flow correct:

1. dead key produces preedit
2. logical shortcut matches and consumes the keydown
3. Linux platform cancels compose / IME state
4. any existing marked dead-key text is deleted
5. any stale Wayland IME follow-up events are suppressed
6. no literal `~` remains in the editor

## Tests

I added tests at two levels.

### Editor-level tests

In `crates/editor/src/editor_tests.rs`:

- `test_deleting_marked_text_removes_dead_key_character`
- `test_unmarking_marked_text_keeps_dead_key_character`

These tests capture the exact semantic distinction at the heart of the bug:

- deleting marked text removes the visible dead-key character
- unmarking only clears composition state and leaves the character in place

These tests are important because the bug only makes sense once the editor semantics are understood. They protect against future regressions where platform code accidentally switches back to unmarking instead of deleting.

### Linux platform tests

In `crates/gpui_linux/src/linux.rs`:

- `test_should_cancel_preedit_for_consumed_dead_key_shortcut`
- `test_should_not_cancel_preedit_for_text_or_unconsumed_input`
- `test_delete_marked_text_replaces_marked_range_with_empty_string`
- `test_delete_marked_text_skips_when_nothing_is_marked`

These tests cover the Linux-specific policy that caused the bug:

- only consumed non-text keydowns should cancel preedit
- normal text input should not cancel preedit
- unconsumed keydowns should not cancel preedit
- deleting marked text must target the marked range and replace it with `""`

## Why These Tests Are Sufficient

These tests are sufficient for this issue because they cover both halves of the regression:

- the decision logic for when Linux should treat a keydown as a consumed dead-key shortcut rather than text input
- the editor/input-handler behavior that determines whether cancelling that shortcut removes or preserves the visible character

The original bug required both halves to go wrong at once:

- the shortcut path had to fail to cancel text composition correctly
- the marked dead-key text had to be preserved instead of deleted

With the current test coverage:

- if Linux stops recognizing consumed dead-key shortcuts correctly, the `gpui_linux` tests fail
- if the text cleanup semantics regress from delete back to unmark, the editor tests fail

That combination gives focused coverage of the actual regression without needing an end-to-end compositor or ibus integration test.
